# Feature Specification: 修复当日涨跌总金额计算中的正负号处理

**Feature Branch**: `001-fix-sort-calculation`
**Created**: 2026-01-22
**Status**: Draft
**Input**: User description: "当日涨跌额没有将正负号参与运算"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 当日涨跌总金额正确计算和显示 (Priority: P1)

用户查看基金列表时，当日涨跌总金额应准确反映所有基金的盈亏情况，正确处理每只基金涨跌幅的正负号，并在总额计算中体现上涨（正值）和下跌（负值）的实际影响。

**Why this priority**: 这是用户评估整体投资表现的关键指标，计算错误会直接误导用户的投资决策。

**Independent Test**: 可以通过设置已知持仓和涨跌幅的基金组合，手动验证总金额计算是否正确。

**Acceptance Scenarios**:

1. **Given** 持有1000元涨幅+1.25%的基金A和2000元跌幅-0.86%的基金B，**When** 查看当日涨跌总金额，**Then** 显示-¥4.70（12.50-17.20）
2. **Given** 持有多只基金，包含上涨和下跌的，**When** 查看总金额，**Then** 正确反映净盈亏，上涨基金贡献正值，下跌基金贡献负值
3. **Given** 某些基金的涨跌幅数据缺失或无效，**When** 计算总金额，**Then** 只计算有效数据的基金，忽略无效数据

---

### User Story 2 - 涨跌总金额显示格式优化 (Priority: P2)

用户查看当日涨跌总金额时，应该看到清晰的正负号标识，帮助用户快速识别当前是盈利还是亏损状态。

**Why this priority**: 良好的显示格式提高用户体验，减少用户对盈亏状态的误解。

**Independent Test**: 可以通过查看不同盈亏场景下的显示格式是否符合用户预期。

**Acceptance Scenarios**:

1. **Given** 当日总盈利为15.45元，**When** 查看显示，**Then** 显示为"+¥15.45"并使用绿色样式
2. **Given** 当日总亏损为15.45元，**When** 查看显示，**Then** 显示为"-¥15.45"并使用红色样式
3. **Given** 当日盈亏为0元，**When** 查看显示，**Then** 显示为"¥0.00"并使用中性样式

---

### Edge Cases

- 涨跌幅数据格式异常（如包含非数字字符、多个正负号、百分号等）时如何处理？
- 持仓金额为0或负数时的处理逻辑是否正确？
- 极大或极小的涨跌幅数值（如 ±99.99%）计算是否会产生显示问题？
- 所有基金涨跌幅都为0时，总金额显示是否正确？
- 涨跌总金额接近0（如0.001）时的显示精度和舍入处理？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须正确解析涨跌幅字符串中的正负号，将其转换为对应的正负数值进行计算
- **FR-002**: 系统必须在计算当日涨跌总金额时正确应用每只基金涨跌幅的正负值
- **FR-003**: 系统必须将上涨基金（正值）的盈利和下跌基金（负值）的亏损分别计算并求和
- **FR-004**: 系统必须正确处理无效或缺失的涨跌幅数据，不让其影响总金额计算
- **FR-005**: 系统必须以一致的格式显示当日涨跌总金额，包括正确的正负号和货币符号位置

### Key Entities

- **涨跌幅数据**: 表示基金当日表现的百分比值，包含正负号的字符串格式（如 "+1.25", "-0.86"）
- **持仓金额**: 用户在特定基金中的投资金额，用于计算实际盈亏
- **当日涨跌总金额**: 所有基金当日盈亏的汇总金额，正值表示总体盈利，负值表示总体亏损

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 当日涨跌总金额计算准确率达到100%（与手工计算结果一致）
- **SC-002**: 用户能在1秒内看到更新后的当日涨跌总金额
- **SC-003**: 涨跌总金额显示格式清晰易懂，95%的用户能在3秒内判断盈亏状态
- **SC-004**: 减少用户因总金额计算错误导致的错误投资判断相关的支持请求
