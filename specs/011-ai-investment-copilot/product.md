# Leek Fund AI Tab 产品文档：投资驾驶舱

**文档版本**：1.0  
**创建日期**：2026-07-23  
**产品阶段**：概念方案，待拆分需求  
**依赖基础**：现有基金、股票、加密货币、黄金行情与持仓能力；现有 AI 会话、SQLite 与 SSE 能力

## 1. 产品摘要

### 1.1 一句话定义

AI Tab 是一个理解用户真实持仓、能解释变化、发现风险、推演选择并记录决策的个人投资驾驶舱。

它不是在理财应用里嵌入一个通用聊天框，而是将 Leek Fund 已有的行情、持仓、历史走势和本地数据组织成可验证的判断依据，帮助用户完成以下闭环：

> 看懂今天发生了什么 -> 找出真正与我有关的问题 -> 比较不同选择 -> 留下决策依据 -> 在未来复盘。

### 1.2 产品命题

普通行情软件回答“市场发生了什么”，通用大模型回答“理论上可能是什么”。Leek Fund AI 要回答的是：

> 基于我此刻的实际持仓，哪些变化值得我关注，为什么值得关注，我有哪些可比较的选择？

### 1.3 核心价值

1. **减少信息噪音**：只优先展示对用户持仓有影响的信息。
2. **降低理解门槛**：把涨跌、集中度、资产配置和风险指标翻译成自然语言。
3. **提高决策质量**：在行动前提供情景推演、反方观点和风险检查。
4. **积累个人经验**：记录当时为什么做决定，并在未来用真实结果复盘。
5. **建立可信 AI**：每个关键数字可追溯，每个推断明确标注，数据不足时不猜。

## 2. 背景与机会

### 2.1 当前产品基础

Leek Fund 当前已经具备以下可复用能力：

- 基金分组、自选列表、实时估值、历史走势、收益区间、资产配置、持有人结构和基金经理等数据。
- 股票、加密货币、黄金的行情与持仓记录。
- 基金、股票、加密货币的持仓金额或份额数据，黄金复用加密资产持仓结构。
- 本地 SQLite 持久化和数据恢复机制。
- AI 会话、消息历史、SSE 流式输出与多 Agent 扩展骨架。
- OpenAI、Claude 和 OpenAI 兼容服务的配置雏形，以及组合分析、每日报告命令雏形。

### 2.2 当前 AI Tab 的局限

现有页面主要完成“创建会话、发送文字、流式显示、保存历史”这一技术闭环，但尚未形成投资产品闭环：

- 页面是空白聊天态，用户需要自己想出问题。
- 对话没有明确携带当前基金、分组或全组合上下文。
- 当前 SSE 对话 Agent 仍返回模拟回声，真实模型命令与流式会话链路尚未统一。
- AI 给出的数字没有来源、数据时间和计算口径。
- 没有组合诊断、情景推演、提醒、决策记录与复盘等任务型能力。
- 没有明确区分“事实”“代码计算结果”“AI 推断”。

### 2.3 产品机会

Leek Fund 的优势不是拥有最多新闻或最快行情，而是掌握了用户主动维护的本地持仓。AI Tab 应围绕这份私有上下文建立差异化，让用户每天打开一次就能快速回答三个问题：

1. 今天我的资产发生了什么？
2. 其中什么值得我处理，而不是只值得我焦虑？
3. 如果要调整，我应该先比较哪些方案？

## 3. 用户与核心场景

### 3.1 目标用户

主要用户是持有多只基金，并可能同时关注股票、加密货币或黄金的个人投资者。他们不一定熟悉专业指标，但愿意记录持仓，希望减少频繁盯盘和情绪化操作。

### 3.2 用户痛点

- 资产分散在不同 Tab，缺少统一视角。
- 知道今天涨跌，却不知道组合为什么涨跌。
- 基金名称不同，但底层风格或风险可能高度重叠。
- 市场信息很多，不知道哪些与自己有关。
- 想调仓时容易只看近期涨跌，没有比较调整前后的结构变化。
- 很少记录买入理由，之后无法判断当初是判断正确还是运气好。

### 3.3 高频使用时刻

| 使用时刻 | 用户问题 | AI Tab 的回应 |
| --- | --- | --- |
| 每日首次打开 | 今天怎么样？ | 生成 30 秒可读的持仓简报 |
| 某资产异常波动 | 为什么跌这么多？ | 展示事实、相关数据、可能原因和未知项 |
| 准备加仓或减仓 | 调整后会怎样？ | 比较多个情景下的配置和集中度 |
| 新增一只基金前 | 它和我已有的重复吗？ | 对比收益、波动、资产配置与已持有资产 |
| 市场剧烈波动 | 我最需要担心什么？ | 按影响金额和风险暴露排序，而非按新闻热度排序 |
| 一段时间以后 | 上次的判断对吗？ | 回放当时数据、理由、触发条件和后续结果 |

## 4. 产品原则

### 4.1 先给结论，再允许追问

首页必须主动呈现“今天最重要的三件事”，不能要求用户面对空白输入框自己发明问题。

### 4.2 先计算，后解释

总资产、涨跌额、配置比例、集中度、收益率和情景结果必须由确定性代码计算。大模型只能解释结果、组织语言、补充风险视角，不能自行心算关键数字。

### 4.3 事实与推断分层

每条重要结论必须属于以下类型之一：

- **事实**：直接来自行情、持仓或历史数据库。
- **计算**：由应用根据公开公式计算。
- **推断**：AI 基于有限证据作出的解释。
- **未知**：当前数据不足，无法可靠判断。

### 4.4 影响优先于热度

风险和事件按“对用户持仓的潜在影响”排序，不按市场讨论热度排序。

### 4.5 建议必须可比较

不输出单一的“你应该买或卖”。涉及调整时，至少提供“保持不变”和一个替代情景，并展示各自代价。

### 4.6 本地优先、出站透明

持仓和对话默认保存在本地。使用云模型时，发送前明确展示将上传的数据范围；使用本地模型时，不应发生隐式外部请求。

### 4.7 AI 不替用户执行交易

AI 可以分析、模拟、建立提醒和生成决策草稿，但不直接连接券商下单，不承诺收益，不将推测包装成事实。

## 5. 信息架构

### 5.1 一级结构

AI Tab 内部设四个任务视图：

1. **今日**：每日简报、异常提醒、待复盘事项。
2. **组合**：资产配置、集中度、盈亏贡献、风险体检。
3. **研究**：单资产解读、多资产比较、情景推演。
4. **复盘**：决策日记、触发条件、历史报告和结果回看。

对话不是独立的第五个页面，而是贯穿四个视图的交互层。用户在任何结论上点击“继续问”，输入框都会自动携带当前上下文。

### 5.2 桌面布局草图

```text
+--------------------------------------------------------------------------------+
| AI 投资驾驶舱     今日  组合  研究  复盘        数据更新 14:32   本地模式       |
+------------------+--------------------------------------+----------------------+
| 今日              | 早上好，今天优先看 3 件事          | 本次分析依据         |
| 组合体检          |                                      |                      |
| 情景推演          | 1. 组合今日 -1.24%，主要拖累...     | 持仓快照 14:32       |
| 决策日记          | 2. 单一资产占比达到 31%...          | 行情 14:31           |
|                  | 3. 一条旧决策达到复盘日期...        | 历史净值 至 07-22    |
| 最近报告          |                                      | 缺失：交易流水       |
| · 7 月 23 日      | [查看贡献] [模拟调整] [加入提醒]    |                      |
| · 7 月 22 日      |                                      | 结论类型             |
|                  | 对话与分析过程                       | 事实  计算  推断     |

| 上下文：全组合 / 截至 14:32    问我的持仓...                  [发送]            |
+--------------------------------------------------------------------------------+
```

### 5.3 页面区域

#### 顶部状态栏

- 当前模型与运行方式：本地模型或云模型。
- 行情、持仓和研究数据的最近更新时间。
- 一键刷新数据，但不在生成过程中偷偷替换分析快照。
- 隐私入口，查看本次请求会发送哪些字段。

#### 左侧任务栏

- 四个一级视图。
- 最近生成的日报、组合体检和研究记录。
- 待复盘数量与未读提醒数量。
- 会话历史折叠在“最近分析”中，以任务标题呈现，不暴露技术性的 UUID。

#### 中央工作区

- 展示结构化报告、图表、证据和对话。
- 结论按影响程度排序。
- 每个结论都可继续追问、加入提醒、进入情景推演或写入决策日记。

#### 右侧证据栏

- 当前回答使用的资产范围。
- 每项数据的来源和时间。
- 计算公式或指标定义。
- 当前缺失的数据及其对结论的影响。
- 窄窗口下折叠为“分析依据”抽屉。

#### 底部输入区

- 支持多行输入与停止生成。
- 以标签展示当前上下文，例如“全组合”“基金 001632”“报告 2026-07-23”。
- 用户可移除上下文，避免误把整个持仓发送给云模型。
- 提供少量与当前页面相关的快捷问题，不展示大而空泛的提示词广场。

## 6. 核心功能设计

### 6.1 今日简报

#### 用户价值

用户无需逐个打开资产 Tab，即可在 30 秒内理解今日组合表现和注意事项。

#### 简报结构

1. **一句话结论**：今天整体表现以及主要驱动因素。
2. **盈亏贡献**：按基金、股票、加密货币、黄金拆分贡献。
3. **最大影响项**：按持仓涨跌金额排序，而非只按涨跌幅排序。
4. **异常与风险**：集中度变化、极端波动、数据过期或缺失。
5. **今天不必做什么**：当没有明显异常时，明确告诉用户无需因普通波动采取行动。
6. **待办事项**：到期提醒、待复盘决策、数据需要补全的持仓。

#### 交互

- “为什么”展开贡献拆解和证据。
- “换个角度”切换为风险、收益或资产类别视角。
- “生成收盘版”将最新数据另存为独立的收盘快照，不覆盖盘中报告。
- 报告可固定、复制 Markdown 或导出，MVP 先支持复制。

### 6.2 组合 X 光

#### 分析维度

- 资产类别占比：基金、股票、加密货币、黄金。
- 单资产集中度：最大持仓占比、前五大持仓占比。
- 分组集中度：按照用户的基金分组理解用途或策略。
- 盈亏贡献度：谁贡献了收益，谁制造了波动。
- 基金风格重叠：基于可得的资产配置、收益相关性和后续接入的重仓数据估计。
- 数据健康度：缺失份额、成本、行情或历史长度不足。

#### 输出方式

组合体检只输出最多三个高优先级问题。其余内容放在“全部发现”中，避免制造风险焦虑。

每个问题包含：

- 发生了什么。
- 为什么与当前用户有关。
- 影响了多少持仓或资产占比。
- 数据证据和更新时间。
- 保持不变会怎样。
- 可以进一步比较的情景，而不是直接的买卖指令。

### 6.3 “为什么涨跌”解释器

#### 触发入口

- 今日简报中的某个资产。
- 基金、股票、加密货币或黄金详情页中的“问 AI”。
- 用户自然语言提问，例如“为什么今天新能源基金都在跌？”

#### 回答模板

```text
结论：目前可以确认的直接原因是什么。

已确认事实：
- 行情与持仓数据。
- 同类资产的同步变化。

可能解释：
- 按证据强弱排序，并标注“推断”。

还不知道：
- 缺少哪些新闻、公告或成分数据，因此不能下结论。

对我的影响：
- 影响金额、组合占比和是否改变原有风险结构。
```

没有接入可靠资讯源时，产品不得生成具体新闻或事件作为原因。此时应诚实回答“当前只能确认价格和组合影响，无法确认事件原因”。

### 6.4 自然语言问持仓

用户可以像查询数据库一样询问自己的资产：

- “我今天亏损最多的三项是什么？”
- “哪些基金没有填写持仓份额？”
- “加密货币占总资产多少？”
- “过去 30 天哪两只基金走势最接近？”
- “找出单日跌幅超过 3%，但仓位低于 5% 的资产。”

实现上先将问题转换为受控工具调用，再由 Rust 查询和计算，最后让模型解释结果。模型不能直接生成并执行任意 SQL。

### 6.5 资产对比与研究

#### 使用场景

用户想新增一只基金，或在两只相似资产之间做选择。

#### 对比内容

- 基本身份与数据时间。
- 不同区间收益。
- 波动与回撤，前提是历史数据长度足够。
- 资产配置差异。
- 与当前组合的走势相关性。
- 加入组合后的占比和集中度变化。
- 数据不足项。

#### 产品约束

“更适合”必须依赖用户明确设置的目标或约束，例如“降低波动”“控制加密资产占比”或“避免同类基金重复”。没有目标时只展示差异，不替用户定义优劣。

### 6.6 情景推演实验室

#### 一句话定义

在真正调整前，先用虚拟方案查看组合结构如何变化。

#### 支持的情景

- 某资产增加或减少指定金额。
- 新增一只尚未持有的资产。
- 将某资产占比调整到目标值。
- 在两项资产之间转移指定金额。
- 设置市场假设，例如“加密货币下跌 15%”“权益基金整体下跌 5%”。

#### 展示结果

| 指标 | 当前组合 | 情景 A | 情景 B |
| --- | ---: | ---: | ---: |
| 总资产 | 确定性计算 | 确定性计算 | 确定性计算 |
| 最大单项占比 | 31% | 26% | 29% |
| 前五项占比 | 72% | 67% | 69% |
| 假设冲击金额 | - | -¥X | -¥Y |
| 数据可信度 | 完整 | 部分历史不足 | 完整 |

AI 负责解释方案之间的取舍，不负责篡改计算结果。所有情景只保存在本地，不直接写入真实持仓；用户明确确认后，才可在后续版本中将情景数据填入持仓编辑表单。

### 6.7 智能提醒雷达

#### 提醒类型

- 价格或涨跌幅达到阈值。
- 持仓占比超过阈值。
- 单日持仓影响金额超过阈值。
- 数据超过指定时间未更新。
- 某个决策的验证条件成立或复盘日期到达。

#### AI 的作用

AI 将多个同时触发的底层规则合并成一条可读提醒，并解释为什么与用户有关。规则判断必须在本地由代码完成，即使模型不可用也能触发。

#### 降噪规则

- 默认按资产和原因聚合。
- 同类提醒在冷却期内不重复弹出。
- 只保留“需要关注”“仅供知晓”两个等级，MVP 不使用夸张的危险等级。
- 用户可查看是哪条规则触发，并一键暂停。

### 6.8 决策日记与反情绪复盘

#### 为什么重要

行情分析很容易变成更高效地追涨杀跌。决策日记让 AI 从“给答案”转为“帮助用户保持前后一致”。

#### 记录内容

- 决策：观察、买入计划、减仓计划、保持不变或取消原计划。
- 当时理由：用户原话优先，AI 只协助整理。
- 关键假设：这项判断依赖什么成立。
- 反方观点：什么证据可能证明自己错了。
- 验证条件：价格、比例、日期或文本条件。
- 复盘日期。
- 当时组合与行情快照。

#### AI 追问

创建日记时最多追问两个高价值问题：

1. “什么情况出现时，你会承认这个判断不成立？”
2. “这次操作是为了收益、降低风险，还是满足短期资金需求？”

#### 复盘输出

复盘时分开评价“决策过程”和“结果”。短期赚钱不自动等于过程正确，短期亏损也不自动等于判断错误。

### 6.9 可执行动作

AI 回答可附带结构化动作，但动作必须分级：

| 动作 | 是否需要确认 | 示例 |
| --- | --- | --- |
| 只读下钻 | 否 | 打开基金详情、查看证据 |
| 本地轻操作 | 是 | 创建提醒、保存报告、写入日记 |
| 修改持仓数据 | 二次确认 | 将情景金额带入持仓编辑 |
| 外部金融交易 | 不支持 | 买入、卖出、申购、赎回 |

AI 必须先展示动作预览。自然语言中的一句“帮我改一下”不能直接修改真实持仓。

## 7. 首页详细方案

### 7.1 首次进入且有持仓

页面自动生成轻量快照，不立即调用大模型。先用本地计算在 300 毫秒目标时间内展示：

- 总持仓金额与今日涨跌金额。
- 各资产类别占比。
- 今日影响最大的三项。
- 数据是否完整、是否过期。

随后按需调用模型生成一句话摘要。模型失败时，本地数据面板仍然可用。

### 7.2 首次进入但没有持仓

空态不展示“有什么可以帮你”，而是给出明确路径：

- 从现有自选中补充持仓。
- 先体验一份仅基于自选、不涉及金额的观察简报。
- 询问基金或市场概念，但明确这是通用问答。

### 7.3 当日第二次进入

默认展示最近一次报告，并标注“此后有 4 项行情已更新”。用户可选择“用最新数据刷新”，旧报告继续保留，保证对话引用的数字不会悄悄变化。

### 7.4 数据异常状态

- 行情部分失败：继续显示成功部分，并隔离失败资产。
- 模型不可用：显示完整本地指标和“AI 解读暂不可用”。
- 持仓不完整：不计算依赖缺失字段的指标，并给出补全入口。
- 云模型未配置：默认提供本地分析能力，不阻塞整个页面。

## 8. AI 回答协议

每次任务型回答由统一结构组成：

```json
{
  "summary": "一句话结论",
  "findings": [],
  "evidence": [],
  "assumptions": [],
  "unknowns": [],
  "actions": [],
  "data_as_of": "2026-07-23T14:32:00+08:00",
  "confidence": "high | medium | low"
}
```

前端渲染结构化字段，不依赖模型自由输出 Markdown 来识别按钮、金额或风险等级。普通聊天文字可以使用 Markdown，但所有关键数字仍需引用工具结果。

### 8.1 证据要求

- 每个金额、比例、排名和收益结论必须绑定数据字段或计算结果。
- 每条证据展示来源、更新时间和范围。
- 同一回答使用固定的 `snapshot_id`，生成过程中行情更新不改变本次答案。
- 推断必须显示“可能”“基于当前数据”等不确定性语言。
- 数据不足时输出未知项，不用常识填空。

### 8.2 推荐回答风格

- 先说与用户有关的影响，再解释市场背景。
- 默认控制在一屏以内，详细计算按需展开。
- 不使用“稳赚”“抄底机会”“必须卖出”等确定性表述。
- 不用免责声明淹没内容，只在涉及决策建议的位置提供清晰边界。

## 9. Agent 与工具设计

多 Agent 是后台分工，不在前台设计多个拟人角色让用户选择。用户始终面对一个统一的“Leek AI”。

### 9.1 Agent 分工

| Agent | 职责 | 允许使用的主要工具 |
| --- | --- | --- |
| 协调 Agent | 识别意图、拆分任务、合并结果 | 上下文读取、任务路由 |
| 组合分析 Agent | 配置、集中度、贡献与数据完整性 | 组合快照、持仓查询、确定性计算 |
| 风险审计 Agent | 寻找反例、过度集中和数据缺口 | 风险指标、历史走势、规则检查 |
| 研究 Agent | 单资产解释和多资产比较 | 资产详情、历史收益、后续资讯检索 |
| 决策教练 Agent | 整理理由、提出反方问题、安排复盘 | 决策日记、提醒、历史快照 |
| 报告编辑 Agent | 将结构化结果压缩成易读简报 | 只读其他 Agent 的结果，不重新计算 |

### 9.2 工具边界

建议提供受控工具，而不是让模型直接访问数据库：

- `get_portfolio_snapshot(as_of)`：获取固定时间的全组合快照。
- `get_asset_snapshot(asset_id, as_of)`：获取单资产数据。
- `get_performance(asset_ids, period)`：返回代码计算的收益与波动指标。
- `compare_assets(asset_ids, dimensions)`：返回结构化差异。
- `simulate_portfolio(changes, shocks)`：返回情景计算结果。
- `list_data_gaps(scope)`：返回缺失或过期字段。
- `create_alert_rule(draft)`：先生成草稿，确认后保存。
- `create_decision_journal(draft)`：先生成草稿，确认后保存。

任何写工具都必须携带一次性确认令牌，避免模型重复调用或误操作。

## 10. 数据设计

### 10.1 当前可直接使用的数据

| 数据 | 能支持的能力 | 当前限制 |
| --- | --- | --- |
| 基金实时估值与历史净值 | 今日贡献、区间收益、趋势比较 | 估值不是最终净值 |
| 基金资产配置、持有人、经理 | 基础画像和结构解释 | 不等于完整持仓穿透 |
| 股票实时行情 | 当日表现和持仓影响 | 当前缺少长期行情库 |
| 加密货币 24 小时行情 | 当日表现和风险提醒 | 当前缺少长期行情库 |
| 黄金行情 | 当日表现和持仓影响 | 当前数据结构与加密持仓复用 |
| 各类资产持仓 | 组合占比和影响金额 | 没有交易流水与现金流口径 |
| 会话与消息 | 连续追问和历史记录 | 当前没有任务、证据与快照关联 |

### 10.2 明确的数据缺口

- 缺少买卖交易流水，因此不能可靠计算已实现收益、资金加权收益和持有期收益。
- 缺少用户现金、负债、投资期限、风险承受能力和资金用途。
- 股票、加密货币和黄金缺少统一的长期历史行情。
- 缺少稳定的新闻、公告、宏观事件和来源引用体系。
- 缺少完整的基金底层重仓穿透数据，不能把风格重叠描述成精确持仓重叠。
- 当前 `holding_amount` 的业务口径需要统一确认是成本金额、当前金额还是用户手工估算金额。

AI 必须根据这些缺口降级输出。产品不得在数据口径未确认前展示“累计总收益”这样的强结论。

### 10.3 建议新增实体

| 实体 | 关键字段 | 用途 |
| --- | --- | --- |
| `portfolio_snapshots` | `id`、`snapshot_at`、`payload`、`data_quality` | 固定一次分析使用的数据 |
| `ai_reports` | `id`、`type`、`snapshot_id`、`content`、`status` | 保存日报、体检和研究结果 |
| `ai_findings` | `id`、`report_id`、`kind`、`severity`、`evidence_json` | 保存可追溯结论 |
| `alert_rules` | `id`、`scope`、`condition_json`、`cooldown`、`enabled` | 本地确定性提醒 |
| `alert_events` | `id`、`rule_id`、`triggered_at`、`snapshot_id` | 提醒历史和去重 |
| `decision_journals` | `id`、`decision`、`thesis`、`invalidation`、`review_at` | 决策与复盘 |
| `ai_request_audits` | `id`、`provider`、`fields_sent`、`created_at` | 云模型出站透明记录 |

所有实体继续使用 SQLite，并包含创建与更新时间。MVP 不要求引入独立向量数据库；知识检索可先使用 SQLite 全文检索和受控的结构化查询。只有在文档数量与语义检索质量证明有必要后，再评估本地向量索引。

## 11. 隐私与模型设置

### 11.1 三种数据模式

1. **纯本地**：本地规则计算 + 本地模型，持仓和对话不离开设备。
2. **金额脱敏云模式**：向云模型发送资产名称、比例和变化，不发送精确金额。
3. **完整云模式**：经用户明确同意后发送完成任务所需的精确字段。

模式切换必须在 AI Tab 顶部可见，不能只藏在设置页。

### 11.2 最小化发送

- 问单只基金时默认不发送全组合。
- 组合分析只发送当前任务需要的字段。
- API Key 使用操作系统安全存储，不写入普通 SQLite 或前端 LocalStorage。
- 保留本地出站审计，用户可查看某次分析发送了哪些字段，不记录 API Key。

## 12. MVP 范围

### 12.1 P0：首个可用版本

目标是让用户每天愿意进入 AI Tab，并能信任它展示的组合摘要。

- AI 首页与今日简报，不再以空白聊天作为首屏。
- 生成统一的全组合快照，聚合基金、股票、加密货币和黄金持仓。
- 本地确定性计算：总持仓金额、今日涨跌金额、资产占比、最大影响项、集中度和数据缺口。
- 基于快照的上下文对话，支持“问全组合”和“问当前资产”。
- 回答展示数据时间、证据类型、未知项和可信度。
- 统一真实模型与 SSE 流式链路，替换当前回声 Agent。
- 模型连接设置：OpenAI、Claude、OpenAI 兼容接口；至少支持 Ollama 作为本地兼容服务。
- 日报保存与最近报告入口。
- 模型不可用时，本地指标仍可正常展示。

### 12.2 P1：形成决策闭环

- 组合 X 光完整视图。
- 两到三项资产对比。
- 情景推演实验室。
- 本地提醒规则与 AI 聚合摘要。
- 决策日记、复盘日期和当时快照。
- 会话与报告按任务管理，可重命名和删除。

### 12.3 P2：形成个人知识系统

- 接入有来源引用的新闻、公告和宏观事件。
- 基金重仓穿透和更准确的组合重叠分析。
- 统一多资产长期行情和相关性分析。
- 导入账单或交易流水，计算持有期与资金加权收益。
- 导入研究文档并进行本地检索。
- 根据历史决策发现用户常见偏差，例如追涨、频繁改计划或忽略失效条件。
- 周报、月报与长期目标偏离提醒。

### 12.4 明确不做

- 不接券商自动交易。
- 不提供收益保证、目标价或无依据的买卖指令。
- 不做匿名社区、跟单和收益排行榜。
- 不在 MVP 引入复杂向量数据库服务。
- 不让多个 Agent 在前台互相辩论制造表演感。
- 不在没有来源时生成“今日新闻原因”。

## 14. 关键验收场景

### 场景 A：每日打开

**Given** 用户至少有两类资产持仓，  
**When** 用户进入 AI Tab，  
**Then** 页面在不等待模型的情况下展示组合金额、今日变化、最大影响项和数据更新时间，并在模型可用后补充摘要。

### 场景 B：追问某个结论

**Given** 今日简报指出某基金是最大拖累项，  
**When** 用户点击“为什么”，  
**Then** 对话自动携带该基金和当前快照，回答区分已确认事实、可能解释和未知信息。

### 场景 C：模型计算错误防护

**Given** 用户询问某资产占组合比例，  
**When** AI 生成回答，  
**Then** 比例来自组合计算工具，前端可定位到分子、分母和快照时间，模型不能覆盖该值。

### 场景 D：情景推演

**Given** 用户计划向某基金增加 10,000 元，  
**When** 用户创建情景，  
**Then** 页面并列展示调整前后的资产占比与集中度，且真实持仓保持不变。

### 场景 E：云模型隐私

**Given** 用户处于金额脱敏云模式，  
**When** 用户发起组合分析，  
**Then** 云请求只包含资产、比例和变化，不包含精确持仓金额，用户可在本地审计记录中查看字段清单。

### 场景 F：模型不可用

**Given** 本地模型未启动或云接口失败，  
**When** 用户进入 AI Tab，  
**Then** 本地组合摘要、风险规则和历史报告仍可使用，页面只将 AI 解读标记为不可用。

## 15. 指标体系

### 15.1 北极星指标

**每周完成一次“有证据的投资任务”的活跃用户数。**

“有证据的投资任务”包括查看完整日报、完成一次组合体检、保存一个情景、创建一条提醒或完成一次决策复盘。单纯发送闲聊不计入。

### 15.2 产品指标

| 维度 | 指标 | 首阶段目标 |
| --- | --- | ---: |
| 激活 | 有持仓用户首次进入后查看完整简报 | >= 70% |
| 效率 | 用户理解今日最大影响项的中位时间 | <= 30 秒 |
| 深度 | 简报后继续查看证据或追问的比例 | >= 35% |
| 信任 | 关键数字具备可追溯证据的比例 | 100% |
| 稳定 | 模型失败时本地摘要可用率 | >= 99% |
| 留存 | 每周至少使用两次 AI 任务的用户比例 | >= 30% |
| 价值 | 报告、情景、提醒或日记的保存率 | >= 20% |

### 15.3 质量指标

- 确定性指标与底层数据计算结果一致率为 100%。
- 无来源的具体事件归因数量为 0。
- 过期数据未标注的关键结论数量为 0。
- 未经确认修改持仓、提醒或日记的次数为 0。
- 相同快照、相同参数的情景计算结果保持一致。

## 16. 风险与应对

| 风险 | 可能后果 | 应对 |
| --- | --- | --- |
| AI 幻觉 | 用户基于错误原因做决策 | 工具计算、证据栏、未知项和来源要求 |
| 数据口径不一致 | 盈亏与比例误导用户 | 先统一持仓金额定义，再开放相应指标 |
| 数据过期 | 使用旧行情生成新结论 | 显示时间、质量状态和刷新入口 |
| 过度提醒 | 用户焦虑或关闭提醒 | 聚合、冷却期、影响优先级和默认克制 |
| AI 诱发频繁交易 | 用户追涨杀跌 | 默认提供保持不变情景和决策日记 |
| 云模型泄露隐私 | 精确持仓暴露 | 本地优先、字段最小化、金额脱敏和出站审计 |
| 多 Agent 成本和延迟 | 首页迟迟无结果 | 本地摘要先行，复杂 Agent 按需调用和缓存 |
| 外部数据不可用 | 原因解释中断 | 降级为价格影响解释，不编造事件原因 |

## 17. 实施建议

### 17.1 第一迭代：可信的组合上下文

- 统一 `holding_amount` 的业务口径。
- 建立 `portfolio_snapshot` 聚合服务和数据质量标记。
- 将已有真实模型配置接入现有 SSE 会话链路。
- 为工具结果定义结构化 Schema 和证据引用。

### 17.2 第二迭代：有用的 AI 首页

- 实现今日首页、本地摘要和最大影响项。
- 实现上下文标签、证据栏与固定快照。
- 实现日报保存、模型失败降级和快捷追问。

### 17.3 第三迭代：从理解走向决策

- 实现组合 X 光和数据缺口修复入口。
- 实现情景推演与方案对比。
- 实现提醒规则、决策日记和复盘。

## 18. 最终产品判断

AI Tab 最容易走向两个低价值极端：一是通用聊天框，什么都能问但和用户资产无关；二是用夸张话术包装几个涨跌榜。真正有长期价值的方向，是让 AI 成为用户资产数据之上的“解释层、审计层和记忆层”。

因此首版最重要的不是接入更多模型，也不是做更多 Agent，而是先做到三件事：

1. 任何结论都知道自己使用了哪一份数据。
2. 任何关键数字都不是模型猜出来的。
3. 用户每次做决定，都能在未来看到当时为什么这样想。

做到这三点，AI Tab 才会从一个功能入口变成 Leek Fund 的核心产品能力。
